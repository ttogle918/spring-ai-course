# 프로젝트 검증 체크리스트

아래는 모듈을 학습(또는 배포)하면서 빠르게 확인할 수 있는 간단한 검증 체크리스트와 추가로 알아두면 좋은 핵심 항목입니다.

## 간단 검증 체크리스트

- [ ] 프로젝트 빌드: 루트에서 `./gradlew :ch14-multi-agent-with-multi-llm:bootJar`가 성공하는지 확인.
- [ ] 환경 변수: 사용하려는 LLM 공급자(OpenAI/Gemini/Ollama)의 자격증명(예: `OPENAI_API_KEY`)을 설정했는지 확인.
- [ ] 조율자 엔드포인트: 애플리케이션 실행 후 조율자 관련 컨트롤러(있다면)에 HTTP 요청을 보내 응답을 받는지 확인.
- [ ] SSE 확인: 조율자에 쿼리를 보내면 SSE 이벤트로 에이전트 진행 상태가 전송되는지 확인.
- [ ] 엔티티 파싱: `PlanAgent`나 `AttractionAgent` 등에서 LLM 응답이 DTO로 정상 매핑되는지(파싱 오류/repair 로그 확인).
- [ ] 로깅: 수리(repair) 시도나 JSON 파싱 오류가 로그에 기록되는지 확인.

## 프로젝트 검증 단계 (권장 순서)

1. 빌드(테스트 포함)→ 2. 환경변수 설정 → 3. 애플리케이션 실행(로컬) → 4. 조율자에 샘플 쿼리 전송 → 5. SSE/응답/로그 확인

## 추가로 알면 좋은 핵심 항목

- **프롬프트 수리 전략**: LLM이 비정형 출력을 했을 때 재요청 시 어떤 안내문구를 붙이는지(예: 'JSON 배열만 출력')를 설계하세요. 자동화된 스키마 검증(예: JSON Schema)을 추가하면 보정 로직이 단순해집니다.
- **공급자별 토큰 메타데이터**: `PromptBenchmarks`에서 수집한 토큰 사용량을 공급자별로 비교 분석해 최적 모델과 비용 정책을 수립하세요.
- **비동기 안정성**: `CompletableFuture` 기반 병렬 실행 시 예외 처리와 스레드 로컬 전달을 견고하게 구현하세요(에미터 누락, 누수 주의).
- **테스트 모킹**: `ChatClient`를 모킹하여 다양한 LLM 응답(정상, 비정상, 지연, 예외)을 시뮬레이션하는 테스트를 만들면 회귀 방지에 도움이 됩니다.
- **보안·데이터 거버넌스**: 외부 웹 검색 결과를 그대로 LLM에 재전달하는 흐름은 개인정보/저작권 위험이 있으므로 필터링/마스킹 정책을 두세요.

---

필요하시면 이 체크리스트를 기반으로 자동화된 검증 스크립트(예: 간단한 curl 테스트 모음)를 생성해 드리겠습니다.
